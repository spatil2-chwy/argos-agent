Face and perception

Presence is continuous, but action is gated

Face recognition

Detect faces, embed with FaceNet, match identities, and maintain a lightweight presence cache.

Depth gate

Rejects fake, far, stale, or low-confidence faces before they can drive interaction.

Attention gate

Checks whether someone is oriented toward the robot before passive listening or greeting opens.

Proactive triggers

Face, navigation, battery, and patrol events pass through a coalescer. The robot only speaks when idle, cooldowns pass, and duplicate events are suppressed.

This is the social layer: seeing someone is not enough; the robot asks whether the signal is real, relevant, and timely.

Embodiment, display, context

The model talks through a situated robot body

Person centering

Owner-turn logic uses face bearing and robot transforms to rotate toward the current speaker.

Gestures + Go2 actions

Idle, listening, and speaking gestures, plus posture tools and expressive action commands.

Runtime-owned display

Shows face state, subtitles, thinking / recording state, and enrollment review UI.

Knowledge + dynamic prompting

Static personality plus live people, memories, robot state, battery, saved locations, and FAISS knowledge results.

The point: the model does not have to guess the room. The runtime curates the room into the prompt and tools.

Logical topic shape

argos/providers/{provider}/resources/{resource}/request/{request_id}
argos/providers/{provider}/resources/{resource}/response/{request_id}
argos/providers/{provider}/resources/{resource}/event/{event_type}
  • Team-friendly: one person can run cody_go2 while another runs puffle_go2; both keep robot-base, realsense_001, and display semantics.
  • Transport-independent: the same logical address maps cleanly to Zenoh, HTTP, WebSocket, or gRPC.
The topic string is the logical address. Each transport can map it into its native form, so the architecture is not tied to one protocol.
